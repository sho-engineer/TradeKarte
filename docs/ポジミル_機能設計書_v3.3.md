## 0. プロダクト定義と用語

**ポジミル**は、日本語の個人裁量FXトレーダー向けAIトレード振り返りアプリです。

チャート画像、本人が事前登録したプレイブック、エントリー理由を使い、AIが次を監査します。

- 自分で決めたルールに沿っていたか
- 記録された根拠が十分か
- ルール逸脱や観察可能な衝動的行動の兆候があるか
- 判断に必要な情報が不足していないか

### ブランドコピー3層

| 用途 | 表現 |
|---|---|
| 問い | そのポジは、エッジか衝動か。 |
| 宣言 | **AIは、結果を知らない。** |
| 体験 | 勝ったのに、怒られる。 |

「エッジ」はマーケティング上の問題提起にだけ使います。単発トレードから統計的優位性を断定しません。

### 単発判定：6分類

DB・API・Zod・TypeScriptでは英語キーを使い、UIだけ日本語表示へ変換します。

| 内部キー | UI表示 | 選択基準 |
|---|---|---|
| `ruleok` | ルール適合 | 重要な不一致がなく、主要な必須条件を確認できる |
| `insufficient` | 根拠不足 | 行動自体は評価できるが、記録されたエントリー根拠が不足している |
| `violation` | ルール逸脱 | 少なくとも1つの重要ルールで`adherence=violated`があり、行動兆候だけを主要因としない |
| `impulse` | 衝動兆候 | 観察可能な行動兆候に具体的根拠があり、確信度が`medium`以上で、レビューの主要論点である |
| `mix` | 混在 | 複数の重要分類が同時に成立し、1分類へ寄せると重要情報が失われる |
| `unknown` | 判定不能 | 画像・時間足・位置・入力不足により主要ルールの適合性そのものを評価できない |

以下を混同しません。

- `insufficient` ≠ `unknown`
- `violation` ≠ `impulse`
- 勝ち ≠ 良い判断
- 負け ≠ 悪い判断

## 2. 認証・アクセス制限

### 2.1 アクセスコード

利用制限はアクセスコード方式だけを使用します。限定URL方式は実装しません。

1. 初回アクセスで`APP_ACCESS_CODE`を入力
2. サーバーで定数時間比較
3. `APP_ACCESS_COOKIE_SECRET`で署名
4. HttpOnly / Secure（本番）/ SameSite=Lax Cookieを発行
5. `/app`配下と`/api/review`の両方で検証
6. Cookieがない、署名不正、有効期限切れの場合は拒否

### 2.2 Supabase Anonymous Sign-In

- メール入力なしで匿名ユーザーを作成
- 匿名ユーザーもSupabase上は`authenticated`ロール
- `auth.uid()`単位でRLSを適用
- `user_id`をリクエスト本文から受け取らない
- サーバーでSupabaseセッションからユーザーIDを確定する
- ブラウザデータを削除すると記録へ再アクセスできない可能性をUIで明記する
- Phase 1でメール・Googleアカウントへのリンクを検討する

### 2.3 サービスロール

`SUPABASE_SERVICE_ROLE_KEY`はサーバー専用です。

用途を次に限定します。

- `ai_review_run`の作成・更新
- 成功レビュー確定RPCの実行
- 結果後入力
- 構造化フィードバック更新
- 入力訂正によるrevision作成
- 失敗ログの更新

`karte`のINSERT・UPDATE・DELETEはクライアントから直接行いません。すべてサーバーRoute Handlerで通常のSupabaseセッションを確認した後、service roleで必要な列だけを書き込みます。

使用前に、必ず通常のSupabaseセッションから`auth.uid()`を確認し、そのユーザーIDだけを保存します。

## 3. プレイブック：自分ルール

### 3.1 自分ルールを設定する機能

あります。ポジミルでは「プレイブック」と呼びます。

初回利用時、保存済みプレイブックが0件なら登録を必須とします。

- システム上の最低数：1個
- UI上の推奨：2〜3個
- 高度な管理画面は作らない
- 入力画面から新規作成可能
- 既存プレイブックを選択可能
- 編集時は既存行を上書きせず新バージョンを作る

### 3.2 1プレイブックの上限

合計最大6条件です。

- 必須条件：最大3
- 見送り条件：最大2
- 損切り・無効化条件：最大1

各条件は1行1条件です。

### 3.3 保存形式

各条件に固定の`rule_id`を持たせます。

```json
{
  "must_rules": [
    {"rule_id":"must_1","text":"上位足が上昇トレンド"},
    {"rule_id":"must_2","text":"サポートへ到達"},
    {"rule_id":"must_3","text":"反発を確認"}
  ],
  "avoid_rules": [
    {"rule_id":"avoid_1","text":"急騰直後は見送る"}
  ],
  "stop_rule": {
    "rule_id":"stop_1",
    "text":"直近安値を明確に下抜けたら無効"
  }
}
```

AIへ渡すときは次へ正規化します。

```json
[
  {"rule_id":"must_1","rule_type":"required","rule_text":"上位足が上昇トレンド"},
  {"rule_id":"avoid_1","rule_type":"avoid","rule_text":"急騰直後は見送る"},
  {"rule_id":"stop_1","rule_type":"invalidation","rule_text":"直近安値を明確に下抜けたら無効"}
]
```

### 3.4 バージョン管理

編集時：

- 元の行は変更しない
- `version = old.version + 1`
- `previous_version_id = old.id`
- 新版を作成
- カルテへ使用時点の`playbook_snapshot`を保存

カルテのフラグ：

- 取引前から存在したものを無編集で使用
  `playbook_created_for_this_review=false`
  `playbook_edited_for_this_review=false`
- 振り返り時に新規作成
  `playbook_created_for_this_review=true`
- 振り返り時に既存を編集
  `playbook_edited_for_this_review=true`

## 4. 入力画面

### 4.1 必須項目

- 取引日時 `trade_at`
- 取引タイムゾーン `trade_timezone`（IANA名）
- チャート画像
- エントリー位置
- クロップ後画像の確認
- 通貨ペア
- 方向 `long | short`
- プレイブック
- エントリー理由
- 入力ソース `recorded_at_time | from_memory`

### 4.2 任意項目

感情セルフタグ：

- `calm`：冷静
- `rushed`：焦り
- `revenge`：取り返したい
- `excited`：興奮
- `anxious`：不安

感情はAIへ送信しません。

### 4.3 取引日時

- UIは`datetime-local`
- 初期値は現在
- `trade_timezone`を別途保存
- DBは`timestamptz`
- 将来の曜日・セッション集計は`trade_at`と`trade_timezone`を使う
- `created_at`を取引時刻の代わりに使わない

### 4.4 入力ソース

- `recorded_at_time`：当時記録していた
- `from_memory`：記憶から入力した

結果を知った後の記憶再構成と、当時の記録を混ぜないために保存します。

### 4.5 結果語チェック

エントリー理由欄の直下に常時表示：

> エントリー時点で考えていたことだけを書いてください。結果や決済後の情報は書かないでください。

結果語候補を検出した場合：

- 自動削除しない
- 確認ダイアログを表示
- 修正または続行を選択
- 続行時は`result_warning_overridden=true`
- 最終送信時点で結果語がなければ`result_word_detected=false`

候補例：

- 利確
- 勝った
- 負けた
- 建値
- 損切りになった
- そのまま伸びた
- 逆行した
- 結果的に
- `+xx pips` / `-xx pips`

## 5. 画像処理とブラインド完全性

### 5.1 画像処理

クライアントで実行します。

1. 画像形式を検証
2. 長辺1280px以下へ縮小
3. JPEG quality 0.85
4. ユーザーがエントリーしたローソク足の右端をタップ
5. `cutX = Math.max(40, Math.round(img.width * frac))`
6. `cutX`より右を完全に削除
7. 追加マージンは入れない
8. クロップ画像右端へ`#5EA8B3`、幅3pxの縦線を描画
9. クロップ画像をプレビュー
10. 確認チェック後だけ送信

UI案内：

> エントリーしたローソク足の右端をタップしてください。タップした位置より右側はAIに送信されません。

確認文：

> この画像に、損益・決済結果・エントリー後の情報が含まれていないことを確認しました。

### 5.2 保存するブラインド情報

- `result_word_detected`
- `result_warning_overridden`
- `image_blind_confirmed`
- `blind_integrity`

成功カルテの`blind_integrity`：

- `clean`
- `warning_overridden`

`unconfirmed`の成功カルテは作りません。確認なしではAPIを呼びません。

Gate 0の厳密なブラインド評価は原則`clean`だけを対象にします。

### 5.3 画像非保存

保存禁止：

- 元画像
- クロップ画像
- base64
- サムネイル
- Storage URL

禁止場所：

- Supabase DB
- Supabase Storage
- localStorage
- IndexedDB
- Vercelログ
- エラー監視
- `console.log`
- `ai_review_run.raw_response`

履歴画面では画像を再表示しません。

## 7. AIレビュー：確定System Prompt

以下を**文字列定数として一元管理し、一字一句変更せず使用**します。

```text
あなたは裁量FXトレーダーの記録を監査するレビュアーです。ユーザーが既に執行したトレードについて、エントリー時点で得られた情報だけを使い、「ユーザー自身が事前に決めたルールに、実際の行動が沿っていたか」を確認してください。

厳守事項:
- 今後の売買シグナル、エントリー、損切り、利確の推奨、将来の価格予測は一切出さない。
- 新しいルールを提案しない。ユーザーが渡したプレイブック条件との照合だけを行う。
- 画像はエントリー地点までにクロップ済みです。画像右端のティール色の縦線をエントリー地点として扱ってください。画像右側のエントリー後の値動きや結果情報は渡されていません。
- 各 rule_check は、入力された rule_id、rule_type、rule_text をそのまま返す。条件を追加、削除、統合、言い換えしない。
- rule_type が required の場合、条件を満たしている observation=met は adherence=compliant、満たしていない observation=not_met は adherence=violated とする。
- rule_type が avoid の場合、見送り条件が発生している observation=met は adherence=violated、発生していない observation=not_met は adherence=compliant とする。
- rule_type が invalidation の場合、無効化条件が発生している observation=met は adherence=violated、発生していない observation=not_met は adherence=compliant とする。ただし画像または文章から確認できない場合は observation=unknown、adherence=unknown とする。
- 画像や文章から直接確認できる事実と、そこから推測される行動兆候を分ける。
- behavior_signals では感情や心理状態を断定しない。「焦り」「FOMO状態」「取り返そうとしている」ではなく、「急騰後の追随」「確認前エントリー」など、画像または文章から観察可能な行動として表現する。
- 各理由には、根拠の出所が画像かユーザー文章かを明記する。
- 画像や入力から行動自体は評価できるが、ユーザーが記録したエントリー根拠が不足している場合は assessment=insufficient とする。
- 画像不鮮明、必要な時間足がない、エントリー位置を確認できないなど、主要ルールの適合性そのものを評価できない場合は assessment=unknown とする。
- assessment=ruleok は、重要な不一致がなく、主要な必須条件が確認できる場合に使用する。
- assessment=violation は、少なくとも1つの重要ルールで adherence=violated があり、behavior_signals を主因としない場合に使用する。
- assessment=impulse は、観察可能な行動兆候に具体的根拠があり、confidence が medium または high で、その兆候がレビューの主要論点である場合に使用する。
- assessment=mix は、複数の重要な分類が同時に成立し、1つに絞ると重要情報が失われる場合に使用する。
- 単発取引から期待値やエッジを断定しない。
- reflection_question は必ず疑問文にする。将来の行動指示を含めない。
- 指定されたJSON Schema以外の文章を返さない。
```

## 8. AI出力JSON Schema

```json
{
  "assessment": "ruleok | insufficient | violation | impulse | mix | unknown",
  "confirmed_facts": [
    "最大4件。画像またはユーザー文章のどちらが出所かを文中に明記"
  ],
  "rule_check": [
    {
      "rule_id": "入力されたID",
      "rule_type": "required | avoid | invalidation",
      "rule_text": "入力された条件文をそのまま返す",
      "observation": "met | not_met | unknown",
      "adherence": "compliant | violated | unknown",
      "reason": "判断理由。画像またはユーザー文章の出所を明記"
    }
  ],
  "behavior_signals": [
    {
      "signal": "観察可能な行動兆候",
      "confidence": "low | medium | high",
      "reason": "具体的根拠。画像またはユーザー文章の出所を明記"
    }
  ],
  "missing_information": [
    "最大4件"
  ],
  "reflection_question": "疑問文を1つ",
  "confidence": "low | medium | high"
}
```

上限：

- `confirmed_facts`：最大4
- `rule_check`：入力条件数と同数、最大6
- `behavior_signals`：最大2
- `missing_information`：最大4
- `reflection_question`：1
- `additionalProperties`：すべてfalse

### 8.1 アプリ側の追加検証

Structured Outputsの後、Zodと追加ロジックで検証します。

- `assessment` enum
- `rule_type` enum
- `observation` enum
- `adherence` enum
- confidence enum
- 配列上限
- 必須項目
- 追加プロパティなし
- `rule_check`件数が入力条件数と同じ
- `rule_id`集合が入力と完全一致
- `rule_id`重複なし
- `rule_type`と`rule_text`が入力値と完全一致
- requiredの評価方向が正しい
- avoidの評価方向が反転している
- invalidationの評価方向が正しい
- `reflection_question`が疑問文
- 行動指示の禁止語を簡易検査
- `stop_reason`が`end_turn`
- `refusal`ではない
- `max_tokens`ではない

検証失敗時、カルテを作りません。

## 10. `ai_review_run`とカルテ保存

### 10.1 全実行を記録

成功・失敗を問わず`ai_review_run`へ記録します。

- `running`
- `succeeded`
- `failed`

失敗段階：

- `request_validation`
- `anthropic_http`
- `network`
- `refusal`
- `max_tokens`
- `structured_output`
- `zod_validation`
- `database`
- `unknown`

### 10.2 保存順

1. アクセスコードを検証
2. SupabaseセッションからユーザーIDを取得
3. リクエストDTOを検証
4. `ai_review_run`を`running`、`karte_id=null`で作成
5. Anthropic APIを実行
6. HTTP、stop_reason、Structured Outputs、Zod、rule mappingを検証
7. 失敗時：
   - `ai_review_run`を`failed`へ更新
   - `karte`を作らない
8. 初回成功時：
   - SQLの`finalize_ai_review_run` RPCでカルテ作成とrun確定を同一トランザクションで行う
   - `is_canonical=true`
9. 同一入力の再実行・モデル比較成功時：
   - 新しいカルテを作らない
   - 既存カルテへrunを関連付ける
   - `is_canonical=false`

同一カルテに`is_canonical=true`は1件だけです。

### 10.3 カルテが持つもの

カルテは確定された監査結果を持ちます。

- assessment
- confirmed_facts
- rule_check
- behavior_signals
- missing_information
- reflection_question
- ai_confidence

カルテはAI実行メタを持ちません。

### 10.4 runが持つもの

- model_id
- prompt_version
- review_schema_version
- status
- failure_stage
- latency_ms
- input_tokens
- output_tokens
- estimated_cost_yen
- parsed_response
- raw_response
- stop_reason
- http_status
- is_canonical
- experiment_id

`raw_response`に画像、base64、APIキー、リクエスト本文を保存しません。

> 自己申告感情とAIが検出した行動兆候を並べて表示しています。実際の感情や因果関係を断定するものではありません。

一致・不一致や原因を自動判定しません。

## 14. 改訂レビュー

入力誤りがある場合だけ新カルテを作ります。

対象：

- エントリー位置ミス
- プレイブック誤選択
- プレイブック入力ミス
- 通貨ペア・方向の誤り
- 画像不鮮明
- エントリー理由の誤記

- 元カルテは変更しない
- 新カルテの`revision_of`へ元カルテIDを保存
- 結果入力だけでは改訂しない
- 将来集計では改訂系列の最新確定版だけを使う
